## GH-131: OMEN 45L fans multiple issues

Source:
- GitHub issue: https://github.com/theantipopau/omencore/issues/131
- Reporter: AndyKr0p

### User-reported behavior

After switching to Performance mode and reboot:
- Turbo mode for fans remains enabled and cannot be switched off.
- One chassis intake fan remains inactive.
- Chassis exhaust fan remains inactive.
- CPU fan becomes inactive.

Detailed context from Reddit follow-up (same issue family):
- User initially could not find uninstall entry because icon/name did not appear clearly in app list.
- After enabling Performance mode, fan behavior became persistent/turbo and inconsistent after reboot.
- User reports BIOS-level fan state also became chaotic, not just in-app telemetry.

### Observed failure sequence (reconstruction)

1. System starts in normal state.
2. User enables Performance mode in OmenCore.
3. Fan response enters high/turbo state.
4. Reboot occurs.
5. Post-boot, multiple fans report inactive and control appears broken until external intervention (OGH/manual paths/BIOS changes).

### Evidence currently missing (required for closure)

- Full startup and shutdown logs covering the transition into Performance mode and the first reboot.
- Capability detection output (model family, selected fan backend, fan controller status).
- BIOS fan monitor screenshots before and after OmenCore apply sequence.
- Confirmation of OGH process/service state during incident.

### Potential files causing concern

- src/OmenCoreApp/App.xaml.cs
  Concern: Desktop detection and experimental-support flow may not enforce runtime write restrictions strongly enough.
- src/OmenCoreApp/Hardware/CapabilityDetectionService.cs
  Concern: Fan backend recommendation for desktops and model-family interpretation may allow unsafe control paths.
- src/OmenCoreApp/Hardware/ModelCapabilityDatabase.cs
  Concern: Desktop entries currently mark SupportsFanControlWmi=true for 25L/30L/35L/40L/45L; policy may be too permissive without stricter safety gates.
- src/OmenCoreApp/Services/FanService.cs
  Concern: ApplyPreset/ApplyMaxCooling/recovery/restore-auto and startup-resume reapply logic can amplify bad state if desktop-specific guard is missing.
- src/OmenCoreApp/Hardware/WmiFanController.cs
  Concern: WMI write semantics, keepalive/countdown ownership, and reset behavior may not match desktop controller expectations.
- src/OmenCoreApp/Hardware/FanControllerFactory.cs
  Concern: Controller selection and mitigation register behavior may assume laptop-style behavior.
- src/OmenCoreApp/ViewModels/MainViewModel.cs
  Concern: Startup restore and cross-surface mode apply triggers may call fan writes during unsafe windows.

### Impact and risk

- Safety risk: High. CPU/chassis fans inactive after control transitions can create thermal hazard.
- Trust impact: Critical. User reports BIOS-level fan behavior becoming chaotic.
- Blast radius: Potentially all OMEN desktop models if laptop fan paths are not hard-gated.

### Likely root-cause hypotheses

1. Desktop safety gate bypass in one or more fan/performance apply paths.
2. Incorrect backend selected on desktop (laptop-oriented WMI/EC assumptions applied to desktop fan topology).
3. Reboot persistence path restoring a stale or incompatible fan state.
4. OGH and OmenCore control-plane contention resulting in invalid post-reboot fan ownership state.

### Proposed solutions

1. Enforce hard desktop block for manual fan writes in all runtime entry points:
- Main apply paths
- Startup restore paths
- Quick Access/tray hotkey actions
- Performance-mode linked fan transitions

2. Add explicit desktop-safe behavior:
- Allow read-only telemetry
- Keep profile/performance toggles only if verified desktop-safe
- Reject fan write requests with warning-level telemetry and user-visible notice

3. Add panic recovery guard on startup:
- If desktop + recent non-default fan state detected, force safe reset attempt and suppress custom writes
- Log one-shot incident marker for support bundle

4. Add OGH coexistence conflict detection:
- Detect active OGH fan ownership indicators
- Prevent write attempts and notify user of exclusive-control requirement

### Validation plan

- Physical desktop validation on OMEN 45L class hardware
- Repro sequence: boot -> Performance mode -> reboot -> verify fan RPM and BIOS monitor consistency
- Negative tests: Ensure no fan-write call executes when desktop gate is active

### Required tests

- Unit: Desktop capability guard tests for all fan write entry points
- Integration: Startup restore does not issue fan writes on desktop
- Regression: Performance mode transitions cannot modify fan EC/WMI path on desktop profile

### Release gate for v3.6.3

- Must pass desktop safety matrix before ship.
- If not fully fixed, ship with desktop fan-control kill switch forced on by default.

### v3.6.3 patch status

- Desktop fan writes are now blocked in model profiles, capability detection, fan controller creation, FanService write paths, shutdown/reset paths, and startup messaging.
- Performance-mode linked fan policy, legacy WMI thermal-policy fallback, and fan calibration/verification now also respect fan-control blocks.
- Targeted regression tests cover desktop capability gating and FanService write suppression.
- Remaining gate: physical OMEN 45L validation and OGH coexistence/recovery evidence.

## GH-132: Fan hunting on HP Omen 15en-100z

Source:
- GitHub issue: https://github.com/theantipopau/omencore/issues/132
- Reporter: dangerweenie

### User-reported behavior

- User removed OGH and other HP fan tools.
- On OmenCore 3.6.2, fan hunting/oscillation occurs across Balanced, Quiet, Gaming, and other plans.
- Report includes model identity summary showing fallback classification:
  - ProductId: 88D2
  - WMI model: OMEN by HP Laptop 15z-en100
  - Resolution: Unknown Legacy Model (family fallback, low confidence)
  - Keyboard model unresolved

Additional details from issue text:
- User explicitly removed OGH and related HP fan software before testing.
- Hunting occurs across multiple profiles, not isolated to one preset.
- Behavior persists with and without overclock settings, indicating primary issue is likely in fan-control control loop and profile mapping, not OC-only.

### Observed failure sequence (reconstruction)

1. Low-confidence fallback model selected (legacy family match, ProductId 88D2).
2. Fan control activated via OmenCore presets.
3. Fan target repeatedly oscillates rather than settling.
4. Oscillation occurs across Quiet/Balanced/Gaming classes, suggesting control loop and cadence mismatch rather than a single preset defect.

### Evidence currently missing (required for closure)

- Full fan command timeline from logs (target percent, actual RPM, write interval).
- Hysteresis settings in effective config at runtime.
- Whether countdown keepalive and deferred verification were active.
- Raw thermal trace during oscillation window.

### Potential files causing concern

- src/OmenCoreApp/Hardware/ModelCapabilityDatabase.cs
  Concern: 88D2 is resolved via broad legacy fallback; fallback capabilities can pick unstable defaults.
- src/OmenCoreApp/Hardware/CapabilityDetectionService.cs
  Concern: Fan backend and confidence-level policy selection for unknown/fallback models.
- src/OmenCoreApp/Services/FanService.cs
  Concern: Hysteresis/dead-zone/ramp timing, deferred verification, recovery writes, and curve-engine cadence under fallback profiles.
- src/OmenCoreApp/Hardware/WmiFanController.cs
  Concern: Keepalive and countdown-extension behavior can contribute to repeated writes and oscillation.
- src/OmenCoreApp/Services/FanVerificationService.cs
  Concern: Verification retries and auto-recovery can produce extra command pressure if telemetry lag is high.
- src/OmenCoreApp/Hardware/HpWmiBios.cs
  Concern: WMI heartbeat/command latency interactions and throttled error behavior under heavy fan polling.
- src/OmenCoreApp/Models/AppConfig.cs
  Concern: FanHysteresis defaults may be too aggressive for unknown legacy thermal response.

### Impact and risk

- High acoustic instability and poor thermal control quality.
- Potential wear/perf instability from repeated fan ramps.
- Indicates low-confidence model profile may be driving unstable control assumptions.

### Likely root-cause hypotheses

1. Family-fallback model profile has incorrect fan scale, cadence, or keepalive assumptions.
2. Curve/hysteresis smoothing insufficient for this hardware response profile.
3. Legacy backend transitions between automatic/manual states are too aggressive.
4. Verification/reapply loop may be over-correcting due to stale readback timing.

### Proposed solutions

1. Add ProductId 88D2 targeted capability profile (conservative defaults):
- Disable aggressive reapply loops
- Increase hysteresis and minimum hold intervals
- Use bounded write cadence for stable target changes

2. Improve anti-hunting fan logic for legacy/fallback models:
- Minimum delta threshold before write
- Ramp-rate limit and dwell timer
- Separate warm-up and steady-state control windows

3. Add low-confidence profile safety mode:
- If model confidence is low, use conservative fan policy and explicit warning in UI/log

4. Add optional diagnostics capture for hunting analysis:
- Commanded percent, measured RPM, temp delta, write timestamps, backend mode

### Validation plan

- Emulated/recorded replay using user-provided timeline where possible
- Hardware validation on 15-en family unit if available
- A/B compare with anti-hunting policy enabled vs disabled

### Required tests

- Unit: Hysteresis and minimum-write-delta behavior
- Integration: No oscillation under stable thermal input trace
- Regression: Existing responsive behavior on validated models remains intact

### Release gate for v3.6.3

- Must include conservative fallback policy for low-confidence legacy mappings.
- Prefer exact 88D2 profile if confidence can be validated quickly.

### v3.6.3 patch status

- Added exact ProductId 88D2 / 15z-en100 conservative legacy WMI V1 profile.
- Direct EC writes, independent curves, GPU power boost, and undervolt are disabled for this profile pending field verification.
- Added conservative curve write policy for 88D2 / unverified legacy WMI profiles: suppresses tiny target deltas below 5%, adds a 10-second non-forced curve-write dwell, and relaxes force-refresh cadence to 60 seconds.
- Remaining gate: field telemetry or hardware validation to tune the anti-hunting thresholds.

## DC-PerKeyRGB: Per-key RGB shown as unsupported on an OMEN Max keyboard

Source:
- Discord report
- Reporter: SimplyCarrying

### User-reported behavior

- User states the keyboard supports per-key RGB but OmenCore reports that it is not supported.
- Screenshot evidence shows the UI card header `Per-Key RGB` with `OMEN Max`, followed by the message that the keyboard does not support per-key control.

### Impact and risk

- Medium UX impact: the app misrepresents a supported hardware capability as unavailable.
- Medium trust impact: users with premium keyboards may assume OmenCore cannot use the hardware they actually own.
- Moderate functional risk: if the model is misclassified as a zone-only keyboard, the app may fall back to reduced lighting control or hide the wrong UI surfaces.

### Likely root-cause hypotheses

1. Keyboard model detection is resolving the device to a zone-based profile instead of a per-key-capable profile.
2. The per-key HID backend is not implemented or not selected, so the service falls back to limited zone/light-bar support and reports that per-key is unsupported.
3. ProductId or model-name mapping for the OMEN Max keyboard is missing or incomplete.
4. A capability flag may be set conservatively in the database even though the physical device supports per-key RGB.

### Observed failure sequence (reconstruction)

1. OmenCore detects an OMEN Max keyboard.
2. Keyboard lighting capability lookup resolves to a non-per-key or generic fallback entry.
3. UI surfaces the `Per-Key RGB` section but displays `This keyboard does not support per-key control.`
4. User concludes the feature is incorrectly blocked despite hardware support.

### Evidence currently missing (required for closure)

- Exact ProductId and WMI model string from the affected keyboard.
- Keyboard diagnostics output showing the resolved `KeyboardModelDatabase` entry.
- Whether the device is a built-in per-key HID keyboard or an external/peripheral RGB device.
- Which backend was selected: WMI BIOS, EC Direct, HID per-key, or fallback.

### Potential files causing concern

- src/OmenCoreApp/Services/KeyboardLighting/KeyboardLightingServiceV2.cs
  Concern: Backend selection logic currently warns that HID per-key is not implemented and may fall back to limited zone/light-bar control.
- src/OmenCoreApp/Services/KeyboardLighting/IKeyboardBackend.cs
  Concern: Defines the per-key capability contract; if a backend never advertises `IsPerKeyRgb`, the UI will correctly but frustratingly hide capability.
- src/OmenCoreApp/Services/KeyboardLighting/KeyboardModelDatabase.cs
  Concern: ProductId/model lookup may not include the exact OMEN Max keyboard entry or may map it to a zone-only configuration.

- src/OmenCoreApp/Hardware/DeviceCapabilities.cs
  Concern: Capability surface exposed to UI may not distinguish per-key from 4-zone keyboards cleanly enough.
- src/OmenCoreApp/Views/LightingView.xaml
  Concern: The UI copy can surface the unsupported message when the resolved capability object says per-key is unavailable.
- src/OmenCoreApp/ViewModels/SettingsViewModel.cs
  Concern: Model identity and keyboard-model labels shown in diagnostics may reflect the wrong keyboard family.

### Proposed solutions

1. Verify the exact keyboard identity first:
- Capture ProductId, WMI model, and system SKU from the affected machine.
- Compare against `KeyboardModelDatabase` and `ModelCapabilityDatabase` entries.

2. Add or correct the keyboard mapping if the hardware is already known to be per-key capable:
- Add the exact OMEN Max per-key model entry.
- Mark the keyboard backend as per-key capable only if that capability is physically verified.

3. Implement or route the correct per-key backend:
- If the keyboard truly uses HID per-key control, the current fallback is insufficient.
- Add the backend implementation or a proper adapter instead of showing a false unsupported state.

4. Make the UI wording more diagnostic:
- Show the resolved model name, ProductId, and backend source when per-key is unavailable.
- Distinguish `not supported by this model` from `backend not yet implemented`.

### Validation plan

- Repro on the same OMEN Max hardware with keyboard diagnostics open.
- Confirm resolved keyboard entry and backend selection before making UI changes.
- Verify the per-key UI only appears when the backend truly advertises per-key capability.

### Required tests

- Unit: Keyboard model resolution for the specific OMEN Max ProductId.
- Integration: Per-key UI visibility matches backend capability.
- Regression: Zone-based keyboards still show the unsupported message correctly.

### Release gate for v3.6.3

- If the hardware is confirmed per-key capable, either expose the backend or clearly mark the feature as pending implementation rather than unsupported.

## DC-Victus16-8BD4: Startup Fans Idle Until Emergency / Fan Stuck at High RPM

Source:
- Discord report from Mr.Carrot.
- Logs: `OmenCore_20260519_174426.log`, `OmenCore_20260519_174426(1).log`, `OmenCore_20260519_232718.log`.

### Evidence from logs

- Device resolves as `Victus by HP Gaming Laptop 16-s0xxx`, ProductId `8BD4`, BIOS `F.30`.
- WMI fan control is selected while direct EC fan control is disabled by the model profile.
- Startup fan RPM readback returns null while fan level readback reports `Fan1=0, Fan2=0`.
- Preset restore repeatedly fails because WMI `SetFanMode(Default)` succeeds but EC register `0x95` reads `0x44`, so OmenCore rejects the preset.
- Several auto/max reset paths send `SetFanLevel(0,0)` on V1 firmware before the user reports fans not starting until emergency thermal protection.
- Thermal protection eventually boosts fans at high temperature, then repeated curve writes/keepalive can hold fans around levels `28-30`, matching the reported high-RPM stuck behavior.

### v3.6.3 patch status

- WMI-only or unverified profiles now use non-strict EC fan-mode readback verification.
- Conservative WMI profiles skip V1 `SetFanLevel(0,0)` auto floor clear to avoid manual-zero fan stops.
- Existing V1 transition hints are retained where they help leave Performance/Max state without forcing zero.
- OSD initialization is now idempotent and lazily initializes the overlay if show is requested before the overlay exists.
- Startup now initializes OSD separately from the broader global hotkey registration path, so startup-minimized launches are less likely to need a settings visit before OSD becomes available.
- Remaining gate: boot-minimized field validation on Victus 16-s0xxx after cold boot and after a long gaming session.

- Include capability confidence, selected fan backend, hotkey reject reasons, and fan write cadence summary in support bundle.

- Publish known-risk matrix and safe fallback behavior in 3.6.3 changelog.
