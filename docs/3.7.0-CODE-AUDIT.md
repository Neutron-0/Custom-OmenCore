## How The Codebase Works

OmenCore is split into a few main runtime surfaces:

- `src/OmenCoreApp` is the Windows WPF application. `App.xaml.cs` owns startup, single-instance behavior, global exception handling, DI registration, tray initialization, and the main window lifecycle.
- `MainViewModel` is the broad presentation hub. It wires the dashboard, fan, general, system-control, lighting, hotkey, tray, OSD, and runtime coordinator paths together.
- `HardwareMonitoringService` is the unified telemetry producer. It reads samples from `IHardwareMonitorBridge`, adapts polling cadence for visible UI, tray, overlay, low-overhead, and suspend states, then broadcasts `SampleUpdated`.
- `FanService` owns fan control and fan telemetry. It maintains active curves, preset application, thermal protection, fan write serialization, RPM smoothing/readback state, and fan command history for diagnostics.
- `PerformanceModeService`, `PowerAutomationService`, and `Runtime*Coordinator` classes own mode application, linked fan/performance behavior, hotkey dispatch, EC serialization, and polling cadence decisions.
- `DashboardViewModel` and `GeneralViewModel` are presentation projections over the monitoring stream. The 3.7.0 work is making these projections throttled and visibility-aware so telemetry cadence does not equal redraw cadence.
- `src/OmenCore.HardwareWorker` hosts LibreHardwareMonitor out of process and communicates with the WPF app over a named pipe. This isolates sensor-driver crashes and lets the main app recover.
- `src/OmenCore.Linux` is the Linux CLI and daemon. Commands call Linux hardware adapters for hp-wmi, ACPI platform-profile, hwmon, sysfs, and legacy `ec_sys` paths.
- `src/OmenCore.Avalonia` is the cross-platform GUI front end. It shares concepts with the WPF app, but it is not yet feature-parity with the mature Windows surface.

The healthy design direction is clear: hardware reads/writes belong in services and controllers, runtime decisions belong in coordinators, and view models should project already-owned state into UI-friendly properties without performing extra hardware work.

## What Could Be Improved Next

### 1. Reduce `MainViewModel` size and ownership

`MainViewModel` still acts as a large orchestration point. It knows about startup restoration, profile cycling, tray/OSD, deferred page view-models, fan mode mapping, and runtime state projection.

Recommended direction:

- Continue moving startup restore policy into a dedicated startup/runtime restoration coordinator.
- Move tray/OSD action formatting and event-history decoration into smaller presentation helpers.
- Keep `MainViewModel` as a composition and presentation layer.

Expected benefit:

- Fewer accidental cross-feature regressions.
- Easier focused tests.
- Lower risk when adding model-specific behavior.

### 2. Centralize UI projection budgets

Dashboard and General now contain their own projection gates. That is an improvement, but the policy is still duplicated conceptually.

Recommended direction:

- Introduce a small `UiProjectionCoordinator` or presentation-budget helper.
- Give each surface explicit budgets: visible, hidden, minimized, tray-only, overlay-active.
- Export budget decisions in diagnostics next to runtime UI counters.

Expected benefit:

- Easier tuning for the 0.3 FPS class field report.
- More consistent throttling across Dashboard, General, tray popup, OSD, and charts.

### 3. Continue fan telemetry projection cleanup

`FanService` now preserves stable `FanTelemetry` objects when fan count is unchanged, and diagnostics now export sync, resize, item-update, and property-only sync counts. The next cleanup step is to make the surrounding dashboard/fan-control presentation budget more explicit.

Recommended direction:

- Keep collection resize behavior limited to real fan-count changes.
- Review dashboard and fan-control fan telemetry subscriptions after more physical model validation.
- Use the new resize/property-only ratios during field validation to confirm normal polling produces mostly property-only syncs after initial fan discovery.

Expected benefit:

- Measurable evidence that fan telemetry no longer contributes to collection-reset storms.
- Cleaner fan telemetry subscription behavior across Dashboard and Fan Control.

### 4. Normalize version metadata

Version data is still duplicated across project files, `VERSION.txt`, installer scripts, README release text, and changelogs.

Recommended direction:

- Make `VERSION.txt` or a repo-level MSBuild props file the source of truth.
- Generate or validate project, installer, README, and release metadata from it.

Expected benefit:

- Fewer release-prep mistakes.
- Simpler hotfix branching.

### 5. Make static globals an explicit migration project

`App.Logging`, `App.Configuration`, and `App.ServiceProvider` keep legacy code easy to call, but they also blur ownership and complicate tests.

Recommended direction:

- Keep compatibility shims short-term.
- Move new code to constructor-injected services.
- Add release-gate checks that block new direct dependencies on `App.Logging` outside approved startup/compatibility files.

Expected benefit:

- Better testability.
- Cleaner service lifetimes.
- Fewer hidden startup side effects.

## Optimization Targets

- Dashboard chart histories: continue moving toward fixed presentation ticks and bounded collection mutation.
- Fan telemetry: review field bundles for high collection resize ratios after startup; repeated resizes would indicate unstable fan-count detection or backend readback shape.
- Hardware monitoring samples: verify that all consumers use the shared `SampleUpdated` stream instead of triggering independent reads.
- Diagnostic export: keep expensive probes opt-in and avoid waking lazy providers only to export idle support bundles.
- Linux CLI output: continue the ASCII-safe direction beyond diagnose output where logs are likely to be copied into Discord/GitHub.
- Build artifacts: reduce checked-in generated logs and release byproducts so audits are not diluted by stale output files.

## Release Risks To Keep Watching

- Physical fan behavior cannot be fully verified by unit tests. `8BCD`, `8D2F`, and Linux board `8787` still need real hardware confirmation.
- UI performance must be validated interactively. Counters help, but frame pacing and input latency require real WPF runtime observation.
- The app still contains historical bare-catch debt tracked by release-gate baselines. The guard prevents growth, but cleanup remains valuable.
- Linux and Windows capability databases need conservative defaults because OMEN ProductIds can span different CPU/GPU generations.
