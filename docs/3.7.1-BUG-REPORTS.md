# OmenCore 3.7.1 Bug Reports From 3.7.0 Stable

**Created:** 2026-06-03  
**Source build:** OmenCore v3.7.0 stable  
**Primary field reports:** Jonathan, HP OMEN 16-am0xxx / ProductId `8D2F`; ZeroMentu, OMEN Transcend 14-fb1xxx / ProductId `8E41`; Mr.Carrot, Victus 16-s0xxx / ProductId `8BD4`; Jack Sparrow, OMEN 15-dc1077tx / ProductId `8574`; Fickert, OMEN 16 Max per-key RGB support request  
**Attached logs reviewed:** `OmenCore_20260602_164235.log`, `OmenCore_20260602_164832.log`, `OmenCore_20260602_080108.log`, `OmenCore_20260602_081456.log`, `OmenCore_20260603_185148.log`, `omencore-diagnostics-2026-06-03-224714.zip`, `OmenCore_20260523_150618.log`

## Summary

This document tracks bugs and regressions identified after the v3.7.0 stable release. The initial 3.7.1 focus is the verified `8D2F` OMEN 16-am0xxx path on Windows 11 25H2 with Intel Core Ultra 7 255H and RTX 5070 Laptop GPU, the `8E41` OMEN Transcend 14-fb1xxx Windows field report with Intel Core Ultra 7 255H and RTX 5060 Laptop GPU, and the `8BD4` Victus 16-s0xxx AMD report with Ryzen 7 7840HS, Radeon iGPU, and RTX 4060 Laptop GPU.

The logs confirm exact ProductId resolution, WMI V1 fan/profile control, PawnIO availability, disabled direct EC fan writes, disabled independent fan curves, and disabled model-level undervolt support for the main 2024/2025 reports. Older legacy evidence for `8574` shows a different class of risk: WMI BIOS is present but command validation fails, so broad legacy EC defaults can expose controls that accept UI input while the hardware effect is weak or missing. The user reports still show several OGH parity gaps around profile performance, fan ramp-down, temperature authority, Eco mode, Quick Access behavior, legacy fan control, and OMEN Max per-key keyboard RGB parity.

## Confirmed From Logs

- ProductId `8D2F` resolves to `OMEN 16-am0xxx (8D2F)` with high-confidence exact identity.
- Fan backend is WMI BIOS, Thermal Policy V1, max fan level 55.
- Direct EC power-limit writes are skipped because the model profile marks direct EC fan/power writes unsafe.
- WMI thermal-policy fallback is applied for Balanced/Quiet/Performance modes when direct EC/MSR limits are unavailable.
- Startup probe briefly reported "Undervolt available via PawnIO" before model refinement disabled it.
- Final capability summary correctly reported `Show Undervolt: No`, but the early probe message and provider initialization remained confusing.
- CPU temperature authority switched between WMI/ACPI and LHM fallback, including frozen/recovered sensor events in the longer log.
- V1 auto-mode handoff skipped `SetFanLevel(0,0)` on `8D2F`, matching the user's report that fans can stay high until manually toggling Custom/Curve.
- Quick Access left-click popup exposed separate fan/performance controls rather than the requested one-click Quick Profiles.
- ProductId `8E41` resolves to `OMEN Transcend 14 (2024) fb1xxx` with exact ProductId but unverified capability confidence.
- `8E41` uses WMI BIOS fan control, Thermal Policy V1, max fan level 55, and one fan zone per model database.
- `8E41` runtime probe reports null fan RPM readback but valid WMI fan level readback.
- `8E41` final capability summary reports `Show Fan Curves: No`, but the UI/session still loaded and applied saved curve presets such as `Extreme`.
- `8E41` logs repeatedly show conservative WMI V1 auto handoff skipping `SetFanLevel(0,0)` after Auto/Default and Max reset.
- `8E41` temperature authority flaps between WMI/ACPI and LHM fallback, with frozen CPU/GPU temperature warnings.
- `8E41` logs register Win+F12 and Ctrl+Shift hotkeys; Win+F12 is observed working, but no Fn+F12 OMEN-key event is observed in the logs.
- `8E41` logs show experimental firmware Fn+P profile cycle being toggled, but no confirmed profile-cycle event follows.
- ProductId `8BD4` resolves to `HP Victus 16-s0xxx AMD`; v3.7.0 diagnostics reported a model-name/low-confidence path even though an exact ProductId candidate was present.
- `8BD4` uses WMI BIOS fan control, Thermal Policy V1, max fan level 55, and direct EC fan writes disabled.
- `8BD4` v3.7.0 fan controller used conservative handoff with "no V1 manual-zero floor clear", matching the report that fans can stay stuck at max after a long gaming session.
- `8BD4` diagnostics confirm a hybrid AMD Radeon iGPU plus NVIDIA RTX 4060 Laptop GPU setup.
- The attached Windows event shows `OmenCore.HardwareWorker.exe` terminating with `AccessViolationException` inside LibreHardwareMonitor AMD ADL frame metrics (`ADL2_Adapter_FrameMetrics_Get`).
- ProductId `8574` / OMEN 15-dc1077tx on OmenCore 3.6.3 resolved through a low-confidence legacy family fallback even though WMI model was `OMEN by HP Laptop 15-dc1xxx`, BIOS was `F.29`, CPU was i5-9300H, and GPU was RTX 2060.
- `8574` WMI BIOS initialization repeatedly found the HP WMI interface but command validation failed with empty system data responses, causing monitoring to fall back to LHM/ACPI and fan control to fall back to PawnIO direct EC.
- The 3.6.3 capability summary exposed `EcDirect`, fan curves, independent curves, undervolt, and EC power-limit style controls on `8574`, but the user later reported that in the new stable release only fan cleaner had any visible effect and even fan cleaner stayed low rather than reaching maximum.
- The `8574` fan cleaner cycle wrote legacy EC fan registers (`0x2C`/`0x2D` = `0x64`, boost register `0xEC` enabled, manual control `0x62` enabled) and logged success, but the follow-up indicates logged EC success is not enough to prove full fan authority on this board.
- Reddit support report from Fickert: OMEN 16 Max owner wants OmenCore performance benefits but keeps switching back to OGH because built-in per-key keyboard RGB is unavailable.
- Current OMEN Max code state: `8D87` / ak0xxx and `8D41` / ah0xxx are mapped as per-key-capable hardware, but `KeyboardMethod.HidPerKey` is not implemented yet and the UI intentionally shows a backend-unavailable message instead of opening a non-working editor.

## Bugs

### 1. Quick Access Popup Shows Split Controls Instead of One-Click Profiles

**Status:** Fixed in 3.7.1  
**Impact:** Users had to manually combine fan and performance choices to mimic Quiet, Balanced, or Performance.

Expected:
- Quiet -> Quiet performance + Quiet fans
- Balanced -> Balanced performance + Auto fans
- Performance -> Performance mode + Performance cooling

Fix:
- Left-click Quick Access now exposes one-click Quiet, Balanced, and Performance profile buttons that route through the existing combined quick-profile path.
- Separate Auto, Curve, and Max fan controls remain available for manual fan-only changes.

### 2. Fans May Stay Elevated After Load on `8D2F`

**Status:** Fixed in 3.7.1 software path; needs field validation  
**Impact:** Fans can remain loud after ramping up and only calm down after the user opens fan control and clicks Custom/Curve.

Evidence:
- Logs show conservative WMI V1 auto handoff with manual-zero floor clear disabled on `8D2F`.
- The affected model is now exact/high-confidence, and the report indicates the conservative no-zero policy causes stuck high fan behavior rather than preventing fan stop.

Fix:
- `8D2F` now keeps direct EC writes disabled but opts into WMI V1 auto-mode floor clear after Default/Auto handoff.
- This lets the firmware clear stale manual fan floors and ramp down after load.
- 3.7.1 launch hardening adds `launch-readiness.txt` diagnostics and a Restore OEM Auto fan action so field reports can show the exact recovery path before/after a stuck-fan scenario.

### 3. Undervolt Capability Message Is Confusing on `8D2F`

**Status:** Fixed in 3.7.1  
**Impact:** Users can see an apparent undervolt-capable state even though apply is unavailable and the model database disables undervolt.

Evidence:
- Phase 9 logged PawnIO undervolt availability.
- Phase 11 then disabled undervolt per model database.
- Final summary showed `Show Undervolt: No`, but the earlier message was misleading.

Fix:
- Capability detection now honors model-level `SupportsUndervolt = false` before runtime PawnIO/MSR probing, so diagnostics and UI state agree from the start.

### 4. Lower FPS and Louder Fans Than OGH Across Profiles

**Status:** Tracked for 3.7.1 field parity; not fully fixed yet  
**Impact:** Reported FPS deltas vs OGH:
- Balanced quick profile: about 2% lower FPS
- Performance quick profile: about 6% lower FPS
- Custom profile with Performance mode + Auto fans: about 3% lower FPS

Current understanding:
- `8D2F` direct EC/MSR power-limit writes remain disabled for safety.
- OmenCore uses Windows power plan plus WMI thermal-policy fallback, while OGH may use additional OEM policy or service paths not yet matched.
- Direct PL1/PL2 and NVAPI power policy writes are unavailable/locked in logs.

Next investigation:
- Capture side-by-side OGH and OmenCore WMI policy, GPU TGP, PL1/PL2, CPU package power, and fan level traces for the same Cyberpunk 2077 scene.
- Verify whether OGH Eco/Performance writes additional display, PPAB, or DTT policy state.

### 5. CPU Temperature Runs Hotter and Readout Is Jumpy

**Status:** Tracked for 3.7.1 field parity; not fully fixed yet  
**Impact:** User reports OmenCore CPU temperatures 5-10 C hotter than OGH and unsmooth readout.

Evidence:
- Logs show CPU authority switching between WMI/ACPI and LHM fallback.
- Longer log shows CPU temperature freeze and recovery events.

Next investigation:
- Prefer the stable worker/LHM CPU package authority for `OMEN 16-am0xxx` if WMI/ACPI authority flaps under load.
- Add field logging around accepted/rejected CPU authority transitions for this model.

### 6. Missing Eco Mode

**Status:** Tracked for 3.7.1; not implemented in this first pass  
**Impact:** OGH has Eco mode that limits CPU/GPU and sets display to 60 Hz; OmenCore only exposes Quiet/Balanced/Performance quick profiles.

Open design:
- Eco should likely be a combined profile, not just a performance mode.
- Expected behavior: lower power policy, conservative GPU power, quiet/auto fan handoff, and display refresh set to 60 Hz when available.
- Needs restore behavior when leaving Eco so refresh rate and power state do not surprise users.

### 7. Power Limit Controls Remain Disabled

**Status:** Expected with current safety model; continue documenting clearly  
**Impact:** Users expect PL1/PL2 sliders because PawnIO is installed, but firmware/MSR paths still report locked or unavailable.

Current behavior:
- Direct EC power-limit writes are blocked for `8D2F`.
- Direct PL1/PL2 controls remain unavailable when readback is `0W/0W` or firmware-controlled.
- WMI thermal-policy fallback remains the safe profile path.

Next investigation:
- Determine whether an OEM WMI or DTT policy path can expose OGH-equivalent power behavior without unsafe EC writes.

### 8. `8E41` Profile-Only Fan Path Still Exposed/Applied Curves

**Status:** Fixed in 3.7.1 software path; needs field validation  
**Impact:** User could create/select aggressive curves even though the `8E41` profile disables custom fan curves. The log shows `Show Fan Curves: No`, but later loads and applies an `Extreme` curve.

Evidence:
- `8E41` model profile has `SupportsFanCurves = false` and `SupportsFanControlEc = false`.
- Session log shows `Loaded last applied custom fan curve from config` and later `Preset 'Extreme' curve enabled`.

Fix:
- FanService now strips curve payloads from built-in fan presets on curve-disabled models and applies only the OEM fan profile mode.
- Manual/custom curve presets are rejected on curve-disabled models instead of being saved as active.
- Direct custom curve calls are also blocked by model capability.
- Direct fixed-speed fan writes are blocked by the same capability gate so profile-only systems cannot be driven through manual fan-level paths.
- Fan Control disables the Custom card/editor, hides saved custom presets, and marks the panel as profile-only when `SupportsFanCurves = false`.

### 9. `8E41` Fans/Settings Not Returning To Expected State

**Status:** Fixed in 3.7.1 software path; needs field validation  
**Impact:** User reports fans only responded after Custom/Max, BIOS switching made no difference, and reopen did not keep the expected selected state.

Evidence:
- `8E41` logs show repeated V1 auto-mode handoff skips during Auto/Default restore and Max reset.
- A later startup reads high fan levels (`47,47`) while the UI/runtime restores labels separately.

Fix:
- `8E41` now opts into WMI V1 auto-mode floor clear after Auto/Default handoff.
- Curve-disabled model gating prevents stale curve/manual state from being restored as active on this profile-only Transcend path.
- Startup restore now skips saved custom curve presets and ad-hoc custom curves on profile-only models, falling back to Auto/OEM fan profiles instead.

### 10. `8E41` Fn Hotkeys Do Not Match User Expectations

**Status:** Tracked; not fixed yet  
**Impact:** User reports Fn+P does not switch profile and Fn+F12 does not open OmenCore. Win+F12 works.

Evidence:
- Logs show `Registered hotkey: Win+F12 -> ShowWindow` and observed `Hotkey pressed: Win+F12 -> ShowWindow`.
- Logs show `Experimental firmware Fn+P profile cycle enabled`, but no subsequent profile-cycle action is confirmed.
- Omen key interception starts, but the supplied logs do not show a Fn+F12 OMEN-key event.

Next investigation:
- Add diagnostic logging for raw low-level keyboard hook events around Fn+F12/Fn+P on `8E41`.
- Confirm whether HP firmware consumes Fn+P before Windows, emits a WMI event only, or uses a scan code outside the current allow-list.

### 11. `8E41` Advanced Lighting Effects Still Not Working

**Status:** Tracked; not fixed yet  
**Impact:** User reports advanced lighting effects still fail.

Evidence:
- Keyboard detection resolves `8E41` as per-key RGB.
- Runtime selects V2 WMI BIOS ColorTable backend and notes `Tried: NewWmi2023`.

Current understanding:
- Static/per-key color table writes may be available while advanced effect commands require a newer WMI/HID path or OGH proxy not yet selected.
- Avoid claiming advanced effects for this profile until the actual effect backend is confirmed.

### 12. Fans May Stay At Max After Long Gaming Session on `8BD4`

**Status:** Fixed in 3.7.1 software path; needs field validation  
**Impact:** Victus 16-s0xxx user reports fans becoming randomly unresponsive and sticking at max speed after long gaming.

Evidence:
- Logs show WMI BIOS fan control, Thermal Policy V1, max fan level 55.
- v3.7.0 logs show the WMI fan controller using conservative handoff with no V1 manual-zero floor clear.
- This matches the same stale-floor handoff class seen on other WMI V1 models.

Fix:
- `8BD4` now keeps direct EC fan writes disabled but opts into WMI V1 auto-mode floor clear after Auto/Default handoff.
- Capability notes now include the 2026-06-03 Discord report and expected field-validation path.

### 13. Hardware Worker Can Crash In AMD ADL Frame Metrics on Hybrid `8BD4`

**Status:** Fixed in 3.7.1 software path; needs field validation  
**Impact:** Hardware worker can terminate after a native AMD ADL access violation, reducing telemetry responsiveness and potentially contributing to stale fan/temperature behavior.

Evidence:
- Event log reports `OmenCore.HardwareWorker.exe` terminated by `System.AccessViolationException`.
- Stack points to LibreHardwareMonitor AMD GPU update path: `ADL2_Adapter_FrameMetrics_Get` -> `LibreHardwareMonitor.Hardware.Gpu.AmdGpu.Update`.
- Diagnostics confirm both AMD Radeon iGPU and NVIDIA RTX 4060 Laptop GPU are present.

Fix:
- Hardware worker now pre-quarantines AMD GPU telemetry on hybrid AMD+NVIDIA systems before any AMD GPU `Update()` call.
- NVIDIA GPU, CPU, memory, battery, fan, and storage telemetry continue to run; only unstable AMD ADL-backed iGPU telemetry is skipped.

### 14. `8BD4` Identity Summary Reports Model Pattern Despite Exact ProductId Candidate

**Status:** Fixed in 3.7.1  
**Impact:** Diagnostics showed low-confidence/model-name inferred capability resolution even though the `8BD4` exact ProductId profile exists.

Evidence:
- Identity trace listed both `Name-pattern candidate: HP Victus 16-s0xxx AMD (8BD4)` and `ProductId candidate: HP Victus 16-s0xxx AMD (8BD4)`.
- Effective resolution still reported `Resolution path: Model-name pattern` and `Confidence: Low`.

Fix:
- Model identity summary now reports non-ambiguous exact ProductId matches before model-name pattern matches.
- `8BD4` should now report `Exact ProductId` with medium confidence until the profile is user-verified.

### 15. OMEN 16 Max Per-Key RGB Editor Is Not Available

**Status:** Tracked; not implemented yet  
**Impact:** OMEN 16 Max users with per-key RGB keyboards still need OGH/Light Studio or another controller for built-in keyboard per-key lighting, reducing OmenCore adoption for users who want to avoid OGH.

Evidence:
- Reddit support report from Fickert asks when OMEN 16 Max per-key RGB will be implemented and whether low-resource alternatives such as OpenRGB can control the keyboard.
- OmenCore detects OMEN Max per-key-capable hardware, but `KeyboardLightingServiceV2` currently logs `HID per-key backend not yet implemented`.
- `LightingViewModel` correctly reports that the per-key editor cannot open on the current backend and keeps zone/light-bar lighting available where applicable.

Current answer:
- There is no committed date for the OMEN Max HID per-key backend in 3.7.1.
- OpenRGB can be suggested for external RGB devices, but it should not be presented as confirmed support for HP's built-in OMEN Max keyboard without hardware evidence.
- The public response should be transparent: OmenCore supports the performance/fan path on these models, recognizes the per-key keyboard, and is holding back the editor until the HID protocol is implemented and field-verified.
- 3.7.1 diagnostics now write explicit per-key backend status in `launch-readiness.txt` and `rgb-control-path.txt`.

Next investigation:
- Collect OMEN 16 Max diagnostics with ProductId, BIOS version, active keyboard backend, and whether OGH/Light Studio is installed/running.
- Capture USB/HID device identifiers for the built-in keyboard and compare OGH/Light Studio behavior against OmenCore's current WMI ColorTable fallback.
- Implement and test a dedicated `HidPerKey` backend before enabling the per-key editor for OMEN Max profiles.

### 16. `8574` OMEN 15-dc1077tx Still Reports No Effective Control

**Status:** Tracked; not fixed yet  
**Impact:** User reports the new stable build still does not make features take effect. Fan cleaner is the only visible feature, but it runs at a low fan level instead of maximum.

Evidence:
- Discord report from Jack Sparrow / REVI: OMEN 15-dc1077tx, ProductId `8574`, System SKU `5CD9237S2J`, BIOS `F.29`, i5-9300H, RTX 2060.
- 3.6.3 logs show HP WMI BIOS present but not command-functional, with repeated empty system-data responses.
- 3.6.3 then falls back to PawnIO direct EC and exposes fan curves, independent fan curves, undervolt, and legacy EC performance/fan paths.
- The later 2026-06-05 follow-up says 3.7.0 support still did not make features effective, which means the current `8574` entry should be treated as partial support rather than validated.

Current answer:
- Do not mark `8574` as fully supported yet.
- Keep diagnostics explicit about WMI command failure versus EC availability so users understand why the UI may show fewer controls or why commands may be limited.
- Prefer disabling or clearly labeling any unverified `8574` fan curve/manual controls unless a fresh 3.7.1 diagnostic bundle confirms RPM/level readback after each command.

Next investigation:
- Ask for a 3.7.1 diagnostic export after pressing Restore OEM Auto, Fan Cleaner, Max, Auto, and one performance profile.
- Add/verify fan command history readback for direct EC writes on `8574`, including EC level registers and measured RPM before/after.
- Compare whether this board expects classic `0-55` level writes, percent-style `0-100` writes, or a different legacy EC/WMI path for true max fan.
- Consider a conservative `8574` profile revision if the fresh 3.7.1 bundle confirms direct EC fan curves/manual writes still log success without meaningful RPM change.

### 17. WPF UI Framework Suggestion

**Status:** Tracked as design input; not a 3.7.1 migration target  
**Impact:** Community suggested evaluating WPF UI (`wpfui.lepo.co`) for a cleaner Windows application shell.

Current answer:
- WPF UI may be useful for a later UI modernization pass.
- It should not block 3.7.1, which is focused on field fixes, diagnostics, and safer capability gating.
- Any migration should be staged after 3.7.1 because replacing the shell/theme framework can easily touch navigation, dialogs, styles, resource dictionaries, and accessibility behavior.
