## Release Improvement Themes

### Fan Control UX

- Make direct fan level control visible when the model supports manual duty/level writes.
- Separate "Max", "constant level", "custom curve", and "OEM Auto" in a way that users can understand without Discord guidance.
- Add clearer capability badges for profile-only systems, WMI-only systems, V1/V2 WMI, and EC-disabled safety models.
- Improve fan status feedback when firmware reclaims control, RPM telemetry is stale, or manual requests are being guarded.

### Visual And Accessibility Pass

- Reduce harsh warning colors and avoid white text on bright yellow/orange backgrounds.
- Review dashboard badges, alerts, chart labels, and warning panels for contrast/readability.
- Improve wording for battery, stale telemetry, fan ownership, and missing-driver states.
- Prefer compact operational UI over explanatory clutter, especially on dashboard and fan-control surfaces.
- First pass implemented: softer dashboard/chart palette, readable chart empty states/stats, and a lightweight OmenCore thermal SVG mark for release/docs artwork.

### GUI Workflow Improvements

- Add a clearer "fan-only" path for users who primarily want OmenMon-style fan control.
- Make OSD/FPS overlay capability and limitations explicit in settings.
- Improve Omen key interception diagnostics when HP services, OGH leftovers, or unsupported keyboard paths block interception.
- Make Quick Profile behavior and custom profile handoff easier to inspect.

### Performance And Optimization

- Track memory footprint complaints and identify high-cost dashboard/monitoring/UI paths.
- Review hardware polling cadence, chart history allocations, tray-only behavior, and background services.
- Keep low CPU usage while reducing idle/background RAM where possible.
- Consider a lighter startup mode that defers RGB, charts, or optional integrations until needed.
- First pass implemented: monitoring charts reuse frozen brushes/dash patterns and dashboard window-state subscription tracking now resets correctly on unload.

### Feature Enhancements

- Continue moving common OGH/OmenMon workflows into OmenCore without unsafe model assumptions.
- Improve diagnostic export readability for field validation.
- Expand battery reporting only when true health/capacity data is available.
- Improve exclusive fullscreen OSD and FPS overlay support if technically feasible.

### Release Gating And Core Stability

- Keep v3.8.0 as RC/pre-release until the core control surface is field-validated: fans, fan curves, performance modes, profile cycling, and basic general hotkeys.
- Prioritize model-family stability over extra features when a control path is still regressing.
- Treat repeated "works in one version, breaks in a later version" fan-control reports as release blockers unless the affected model is explicitly scoped as unsupported.
- Add validation notes that make it clear which model families are verified, partially supported, or still experimental.

## BUG-3800-001 - OMEN Max `8D41` Max Fans Do Not Hold

**Reporter:** SimplyCarrying
**Model:** OMEN MAX Gaming Laptop 16t-ah000
**ProductId:** `8D41`
**Version:** 3.7.1
**Severity:** High
**Status:** Fixed locally - field validation pending

### Report

> Max fans don't stay at max speed. Randomly dropping to 100 and 100 and then slowly climbs back up and repeats.

### Log Evidence

- Model identity resolves correctly by exact ProductId with high confidence.
- WMI reports Thermal Policy V1, but OMEN Max is detected and fan commands are forced to V2.
- MaxFanLevel is `100`.
- Max apply initially verifies via fan level around `40-41`.
- During Max hold, the keepalive detects repeated external reset / firmware reclaim patterns:
  - `External fan reset suspected - Max mode re-applied after sustained drop`
  - observed levels include `60/58`, `52/48`, `34/36`, `28/30`, and `0/0`
- Reapply cadence is about 20 seconds in the field log, so the user sees a sawtooth instead of a stable Max hold.

### Working Theory

The v3.7.1 Max maintenance logic waits for two low-telemetry samples before reasserting Max. On `8D41`, the firmware appears to reclaim or partially reset Max quickly enough that the two-sample delay lets fan level visibly fall before OmenCore reasserts ownership.

### Implemented First Fix

- Add model/capability support for aggressive Max hold maintenance.
- For `8D41`, reassert Max on the first confirmed low Max telemetry sample.
- Keep the model on WMI-only fan paths and avoid legacy EC writes.

### Validation Needed

Retest Max mode under load and confirm fan levels/RPM no longer visibly sawtooth.

## BUG-3800-002 - Victus `8BD4` v3.7.1 Fan Reaction Regression

**Reporter:** Mr.Carrot
**Model:** Victus by HP Gaming Laptop 16-s0xxx
**ProductId:** `8BD4`
**Version:** 3.7.1
**Severity:** High
**Status:** Fixed locally - field validation pending

### Report

Initial v3.7.0 report:

> After long gaming session fans are getting stuck at max speed.

v3.7.1 follow-up:

> still not fixed

Later follow-up:

> Now they don't even react at all

Screenshot shows CPU/GPU fan RPM at `200` while System fan mode says `Max`.

### Log Evidence

- Model identity now resolves by exact ProductId (`8BD4`) in 3.7.1.
- Hardware worker AMD ADL crash from 3.7.0 is addressed by v3.7.1 quarantine work, but fan behavior remains unresolved.
- `8BD4` is using WMI V1, MaxFanLevel `55`, and v3.7.1 enabled `AllowV1AutoModeFloorClear`.
- The log repeatedly sends `SetFanLevel(0, 0)` during preset auto handoff and max-mode reset:
  - `V1 auto-mode floor clear: SetFanLevel(0, 0) succeeded`
- The same session later records zero-RPM/duty problems after nonzero and Max requests:
  - `Curve zero-RPM watch: primary fan reports 0 RPM after 68% request`
  - `Curve zero-RPM watch: primary fan reports 0 RPM after 100% request`
  - recovery only occurs after a one-shot wake kick.

### Working Theory

The `8BD4` v3.7.1 override made the V1 zero-floor clear too aggressive for this Victus board. On this firmware, `SetFanLevel(0,0)` can behave like a transient manual stop or stale zero duty state, which matches the user's "fans don't react" and 200 RPM/0 RPM observations.

### Implemented First Fix

- Revert `8BD4` to the conservative V1 handoff path.
- Keep exact ProductId identity and AMD ADL crash quarantine.
- Preserve WMI profile control, but skip `SetFanLevel(0,0)` on this board until a safer floor-clear sequence is field verified.

### Validation Needed

Retest Max, Auto, Performance, and custom curve behavior. Confirm the UI no longer shows 0/200 RPM while Max or high curve requests are active.

## BUG-3800-015 - PawnIO Setup Fails During Windows Installer

**Reporter:** Rave-TZ / Zakerino007
**Source:** Reddit r/omencore
**Version:** 3.7.1
**Severity:** High
**Status:** Fixed locally - installer artifact validation pending

### Report

During OmenCore setup, the installer reaches "Installing PawnIO driver (Secure Boot compatible)..." and PawnIO shows a modal error:

> Please specify either -install or -uninstall

Users asked whether PawnIO can be installed standalone.

### Root Cause

`installer/OmenCoreInstaller.iss` launched the bundled `PawnIO_setup.exe` with `-silent` only. The PawnIO setup accepts dash-style silent flags, but it still requires an explicit operation verb.

### Implemented Fix

- Changed the Inno Setup `[Run]` entry to launch `PawnIO_setup.exe` with `-install -silent`.
- Kept the existing `Check: not IsPawnIOInstalled` guard so upgrades do not reinstall PawnIO unnecessarily.
- Updated release hygiene tests to assert the embedded setup runs from `{tmp}` with the required install verb.

### Standalone Fallback

If a user is stuck on an older installer, they can run the bundled PawnIO setup manually from an elevated terminal:

```powershell
.\PawnIO_setup.exe -install
```

Reboot afterward, then start OmenCore as Administrator.

## BUG-3800-003 - Battery Health Warning Uses Battery Charge

**Reporter:** OOA
**Model:** OMEN 17 db-1000
**Version:** 3.7.1
**Severity:** Medium
**Status:** Fixed locally

### Report

> On an OMEN 17 db-1000 it was giving me an alert that said "your battery health is at 68%" when it's at 68% battery charge while plugged in. It seems to have gone away when it got to 70% battery charge.

Screenshot shows the historical chart labelled `Battery Health`, but the plotted values track charge percentage.

### Root Cause

`HardwareMonitoringService.UpdateDashboardMetrics` populated `BatteryHealthPercentage` from `MonitoringSample.BatteryChargePercent`. The alert threshold then treated any charge under 70% as a degraded-health warning.

### Implemented Fix

- Split dashboard battery charge from battery health.

- Keep historical battery charting as charge history and relabel the UI from Battery Health to Battery Charge.
- Add regression coverage so 68% charge does not create a Battery Health warning.

## BUG-3800-006 - Victus / OMEN RGB Applies To Wrong Surface Or Not At All

- On one `8D87 + 88F7` setup, RGB changes only the light bar and not the keyboard.

The current RGB capability path is not explicit enough about which physical surface is being controlled: keyboard zones, single-color backlight, light bar, WMI ColorTable, V2 EC, or future HID/per-key routes. A successful light-bar write can look like a keyboard RGB failure to the user.

### Needed

- WMI ColorTable writes are no longer restored/undone just because firmware readback is stale or mismatched after the BIOS accepted the write; diagnostics should treat that as accepted/unverified until the intended physical surface is confirmed.
- Restore-defaults now arms brightness before writing colors, matching the safer profile-apply order.
- Added a Lighting-page observed-surface probe so field reports can record whether the safe test color changed keyboard zones, per-key keys, light bar, single backlight, nothing, or another surface; exports now include that persisted observation.

### Still Needed

- Continue avoiding "RGB works" as a single boolean when only one lighting surface is verified; use the observed-surface field report instead.
- Dedicated OMEN Max HID/per-key backend; WMI fallback may still affect a light bar rather than the built-in keyboard on some MAX boards.

## BUG-3800-010 - Start With Windows Toggle Enabled But App Does Not Launch

**Reporter:** efesunger
**Version:** 3.8.0 pre-release context
**Severity:** Medium
**Status:** Fixed locally

### Report

The Settings page shows Start with Windows enabled, but OmenCore does not launch automatically after Windows starts.

### Root Cause

The Task Scheduler XML wrote the executable path as a quoted XML `<Command>` value. In scheduled-task XML, `<Command>` should contain the raw executable path; embedded quotes can make the task exist and appear enabled while the action fails to launch.

### Implemented Fix

- Startup task XML now writes the raw executable path in `<Command>`.
- Added `<WorkingDirectory>` so startup runs from the app folder instead of Task Scheduler's default directory.
- Startup detection now validates the scheduled task XML, rejects disabled tasks and quoted-command tasks, and no longer treats a malformed task as a working Start with Windows setup.
- Administrator detection for task-creation failure now uses `WindowsPrincipal.IsInRole(Administrator)`.

## BUG-3800-013 - General Tab Shows `0W` CPU Power When Sensor Is Unavailable

**Reporter:** Reddit `r/omencore` bug report
**Severity:** Low-Medium
**Status:** Fixed locally

### Report

The General page CPU card can show live CPU temperature, load, and fan RPM while the CPU power row displays `0W`, making an unavailable or zero-returning package-power sensor look like a real measurement.

### Root Cause

The General page bound CPU/GPU power directly to raw numeric watt properties with `StringFormat`. When the backend returned the default scalar value `0`, the UI rendered it as a confident `0W` instead of using telemetry state or placeholder display logic.

### Implemented Fix

- General view model now exposes CPU/GPU power display strings, availability flags, and tooltips derived from telemetry state.
- CPU/GPU power cards now show `--W` when the sensor is unavailable, invalid, inactive, unknown, or returns zero.
- Positive watt readings still render normally.
- General telemetry projection now refreshes when CPU power state changes even if the numeric watt value does not.
- Added regression coverage for zero CPU power display and valid positive CPU power display.
