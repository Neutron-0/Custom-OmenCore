# OmenCore v3.8.0 - Core Controls Next Steps

**Goal:** make fans, RGB, overclocking, and undervolting trustworthy before spending more release budget on secondary features.

---

## Release Priority

### P0 - Must Work Before Stable

- [x] Fan ownership and handoff diagnostics: Max, Direct, Custom Curve, Performance-linked profiles, and Restore OEM Auto now report backend, requested command, capability gates, current RPM/duty snapshot, hold/curve/thermal state, WMI reclaim indicators where available, and recovery guidance in exports. Field validation still needs true before/after hardware readback on priority boards.
- [x] Model capability truthfulness first pass: unsupported/degraded fan paths are gated for desktop, profile-only, WMI-only, MAX-series, `8BD4`, `8C30`, `8DCD`, `878C`, `8600`, `88EE`, and Linux `8BCD` degraded-control cases. Continue adding exact model entries instead of broad family guesses.
- [x] RGB surface diagnostics first pass: readiness and RGB control-path exports report HP keyboard backend/surface state, per-key capability, last apply status, external providers, and known HP/OMEN conflict processes.
- [x] RGB observed-surface capture: added a lightweight probe/observation flow that records what physical surface actually changed (keyboard zones, per-key keys, light bar, single backlight, none, or unknown) and carries it into diagnostics.
- [x] Tuning readback diagnostics: readiness and tuning safety exports show saved startup values, pending/recovery state, performance apply traces, and readback rules; GPU power readback now distinguishes Extended vs Maximum.
- [x] Tuning rollback bundle: added a single safe rollback path that normalizes startup PL1/PL2 to startup-read values when known, resets GPU OC/power boost, clears undervolt offsets, resets AMD power defaults, and reapplies Balanced/Auto where live backends are available.
- [x] Startup restore safety first pass: startup tuning confirmation/recovery state and scheduled-task reliability are logged and exported.
- [x] Startup restore UX split: fan, performance/GPU power, RGB, and tuning startup restore now have separate opt-in category gates under the broad safety switch.

### P1 - RC Quality

- [x] Per-model validation cards for priority boards: `8D41`, `8D87`, `8BD4`, `8C30`, `8DCD`, `878C`, `8600`, `8BCD`, OMEN 17 db-1000, and Victus 15/16 field reports.
- [x] Diagnostics export includes a single "control readiness" report with fan backend, RGB backend/surface, tuning startup state, monitoring/readback state, last command traces where available, and suggested validation actions.
- [x] Field tester script: one page of exact steps for fans, RGB, performance mode, profile cycling, hotkeys, and startup restore.
- [x] OmenMon-Reborn parity report: export maps common OmenMon expectations to OmenCore's safe-control equivalents and field evidence requirements.
- [x] RC validation matrix: `rc-validation-matrix.txt` records local fix status, field-pending status, and promotion evidence per priority model cohort.
- [x] Installer driver setup guard: Windows installer now passes PawnIO's required `-install` verb with the silent flag and release hygiene tests cover the bundled setup invocation.

### P2 - Nice To Have

- [x] FPS overlay improvements: RTSS FPS now falls back to average FPS when instant FPS is zero, shows compact avg/1%/process detail, color-codes FPS quality, and reports RTSS unavailable/waiting states instead of misleading activity labels.
- [x] Extra RGB effects: added built-in Heat Wave and Calm Pulse scenes, routed Wave scenes through provider-supported `effect:wave` instead of Spectrum fallback, and protected the audio-reactive built-in scene from removal.
- [x] More game/profile automation: game profile detection now honors the feature toggle before starting process monitoring, stores profiles in the configured OmenCore config folder, prefers exact executable-path matches over generic executable profiles, suppresses duplicate reapplies for an already-active profile, and lets profiles opt out of default restore on exit.
- [x] Decorative visual work that does not reduce control clarity: added lightweight thermal mark and chart/card polish without expanding release risk.

## Practical Engineering Steps

### Implemented In This Sweep

- Added `core-control-readiness.txt` to Windows diagnostic exports.
- Added `omenmon-reborn-parity.txt` to Windows diagnostic exports so users comparing OmenCore with OmenMon-Reborn can see which probe, calibration, lightweight, EC-hardening, fan, RGB, and OMEN-key expectations are implemented, partial, degraded, or still evidence-gated.
- Added `field-validation-script.txt` to Windows diagnostic exports with exact repeatable steps for fans, RGB surfaces, performance/tuning readback, profile cycling, hotkeys, startup restore, and evidence attachments.
- Added `priority-model-validation-cards.txt` to Windows diagnostic exports with targeted validation cards for `8D41`, `8D87`, `8BD4`, `8C30`, `8DCD`, `878C`, `8BCD`, OMEN 17 db-1000, and Victus 15/16 field cohorts.
- Added `rc-validation-matrix.txt` to Windows diagnostic exports so every bundle states which priority cohorts are fixed locally, field-pending, degraded, or still experimental for the RC gate.
- Added a lightweight `CoreControlReadiness` section to the in-app diagnostic export control.
- The readiness report now summarizes fan backend/write/curve/manual availability, current fan mode, active preset, command history count, and WMI fan ownership properties when available.
- Fan command history now records ProductId/model, write/curve/manual/desktop safety gates, current telemetry snapshot, and raw primary RPM/duty markers with every command.
- The in-app diagnostic export mirrors the enriched last fan command model/gate/readback fields so UI-created support bundles match `core-control-readiness.txt`.
- The RGB section now reports HP keyboard backend/surface state, last apply status, external provider availability, and known HP/OMEN RGB conflict processes.
- Added an RGB observed-surface capture flow on the Lighting page: users can send a safe bright probe color, save what physical surface changed, and include the result in `rgb-control-path.txt`, readiness diagnostics, and in-app support bundles.
- The tuning section now records startup restore flags, saved performance/GPU power/CPU PL/TCC values, undervolt and GPU OC apply-on-startup state, startup recovery requirements, and recent performance-mode apply trace entries.
- Added a safe tuning rollback coordinator and Tuning page action that first normalizes persisted startup tuning state, then best-effort restores fan OEM auto, Balanced performance, Minimum GPU power, startup CPU PL1/PL2, zero TCC, cleared undervolt, reset GPU OC, and AMD power defaults where supported.
- Split startup restore into category gates for fans, performance/GPU power, RGB, and tuning. Legacy configs with the broad restore enabled keep their existing behavior until users edit the category toggles; fresh/disabled configs remain off.
- The monitoring section now records telemetry source, health, cadence, low-overhead mode, last sample age, and transition count for readback confidence.
- General telemetry cards now hide unavailable CPU/GPU power readings as `--W` with state tooltips instead of rendering backend zero/default values as real `0W` measurements.
- FPS OSD now uses a tested RTSS display formatter: instant FPS is preferred, average FPS is used as a safe fallback, frametime is derived when RTSS does not publish one, and unavailable RTSS states stay explicit.
- RGB scenes now include Heat Wave and Calm Pulse built-ins; Wave scenes route to provider capability-filtered `effect:wave` rather than being collapsed into Spectrum.
- Game profile automation now respects the feature toggle at process-monitor startup, resolves exact executable-path profiles before generic executable profiles, avoids duplicate apply events for the same active profile, and carries exited-profile restore policy into the restore/defaults path.
- The hotkey section now records registered and pending hotkeys, physical OMEN-key hook/WMI watcher state, strict-mode and firmware Fn+P profile-cycle flags, and recent never-intercept suppression evidence for field validation.
- Added regression coverage for the new readiness/parity/field-validation/model-card reports and export bundle entries.
- Replaced high-visibility emoji/glyph labels in core views with vector icons and restored virtualization hints for large game-library lists.
- Fixed the bundled Windows installer PawnIO handoff so the optional driver task launches `PawnIO_setup.exe -install -silent` instead of triggering PawnIO's missing-operation modal.
- Replaced mojibake-prone window-control glyphs with ASCII-stable chrome text and logged expected title-bar drag failures instead of swallowing them.

### Fans

- Extend fan command trace detail where possible: command name, requested percent/profile, backend, ProductId, capability gates, current RPM/duty snapshot, and ownership state are now captured; true before/after hardware readback still needs field validation or backend-specific sampling where safe.
- Add model-family validation presets for field testers: Max hold 10 minutes, Direct 40/60/80%, Restore OEM Auto, curve ramp, profile cycle, suspend/resume.
- Keep `8D41`/`8D87` WMI-only. Do not re-enable legacy EC writes for MAX-series boards.
- Keep `8BD4` conservative until Auto -> Max -> Auto and long-session curve tests prove zero-floor handoff is safe.
- Keep `8C30` WMI-policy-first with CPU power-limit UI hidden until Quiet/Balanced/Performance readback proves a safe wattage override.
- Keep `878C` WMI-policy-first until Performance/Balanced/Quiet profile tests prove fan RPM and PL readback respond safely without direct EC writes.
- Keep `88EE` on exact conservative Victus 16-e0194nw identity until field exports prove fan/RGB/tuning support beyond WMI profile and backlight-only routing.

### RGB

- RGB probe flow is implemented with a safe static test color and persisted observed target surface.
- Keep local RGB observations out of the static model database until enough field reports prove a per-model rule.
- Prefer verified backend labels over generic success. "Accepted/unverified" should remain visible when firmware readback is stale or when the physical surface is unknown.
- Keep the HID per-key backend write-only and conservative until real OMEN MAX PIDs and segment behavior are confirmed.

### Overclocking And Undervolting

- Promote the tuning readiness report into a UI panel after the diagnostic data proves useful in field reports.
- Require readback or explicit "BIOS locked / provider unavailable" text after every tuning apply.
- Safe rollback bundle is implemented; next field pass should validate live rollback readback on Intel MSR, NVIDIA NVAPI, AMD SMU/ADL, and WMI fan-control systems.
- Startup restore is now opt-in per category under the broad safety gate; field validation should confirm each category independently after restart.

### Responsiveness And Premium Feel

- Lazy-load heavy tabs: RGB providers, game library scans, diagnostics snapshots, and optional integrations should initialize when opened or when explicitly enabled.
- Replace emoji/glyph text in core controls with vector `Path` icons from `ModernStyles.xaml`; this avoids font fallback, mojibake, and inconsistent sizing.
- Keep high-refresh dashboard/fan visuals allocation-light: freeze brushes, reuse pens/geometries, cap chart history, and avoid layout invalidation on every sample.
- Use virtualization or paging for large lists. Avoid `ScrollViewer` around large `ItemsControl` surfaces unless the item count is small and bounded.
- Decode images at display size and use `CacheOption=OnLoad` for repeated brand/product assets.
